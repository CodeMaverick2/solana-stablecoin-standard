# SSS-1: Minimal Stablecoin Standard

## Overview

SSS-1 is the minimal stablecoin preset in the Solana Stablecoin Standard. It provides essential token management capabilities -- mint authority, freeze authority, on-chain metadata, role-based access control, and per-minter quotas -- without compliance enforcement features such as blacklists, transfer hooks, or token seizure.

SSS-1 is suitable for use cases that do not require regulatory compliance tooling at the protocol level.

**Program ID**: `B1zqgaJkbVzNoMagPyAJdgveArzaTW6fkyk3JtSq1pHs`

---

## Use Cases

- **DAO treasuries**: Internal settlement tokens managed by a DAO multisig.
- **Ecosystem tokens**: Protocol-specific stablecoins for DeFi applications.
- **Testing and development**: Fast prototyping of stablecoin mechanics before upgrading to SSS-2.
- **Internal settlement**: Cross-entity settlement within a consortium.
- **Reward tokens**: Stable-value reward tokens with controlled issuance via minter quotas.

## Token-2022 Extensions

SSS-1 uses a single Token-2022 extension:

| Extension | Purpose |
|-----------|---------|
| **MetadataPointer** | Points to the mint account itself for on-chain metadata storage |

The metadata (name, symbol, URI) is stored directly in the mint account using the Token-2022 metadata extension. This eliminates the need for Metaplex or any external metadata program. The config PDA serves as the metadata update authority.

---

## Initialization

### Mint Creation

The mint is a fresh `Keypair` generated by the client at initialization time. It is **not** a PDA. The client passes the mint keypair as a signer to the `initialize` instruction.

During initialization, the program uses raw CPI (`invoke` / `invoke_signed`) rather than Anchor's CPI wrappers. This is necessary because the set of Token-2022 extensions must be configured dynamically at creation time based on the preset. The extension initialization order is:

1. `create_account` -- allocate space for the mint with the required extensions.
2. `initialize_metadata_pointer` -- point the metadata to the mint account itself.
3. `initialize_mint2` -- initialize the mint with the config PDA as both mint authority and freeze authority.
4. `initialize_token_metadata` -- write the name, symbol, and URI into the mint account.

### Accounts Created

| Account | Type | Seeds | Program |
|---------|------|-------|---------|
| **Mint** | Token-2022 Mint | N/A (Keypair) | Token-2022 |
| **StablecoinConfig** | PDA | `["stablecoin_config", mint.key()]` | Stablecoin |
| **RoleConfig** | PDA | `["roles", config.key()]` | Stablecoin |

### Config PDA as Authority

The `StablecoinConfig` PDA is set as:

- **Mint authority** -- only the program (via the config PDA) can mint tokens.
- **Freeze authority** -- only the program (via the config PDA) can freeze/thaw accounts.
- **Metadata update authority** -- only the program can update on-chain metadata.

No external account ever holds these authorities directly, which means Token-2022 operations like `mint_to` or `freeze_account` can only occur through the stablecoin program's instructions.

## Features

### Mint Authority with Minter Quotas

Tokens can only be minted through the `mint_tokens` instruction by authorized minters. Each minter has an individual `MinterConfig` PDA that tracks their quota:

```
MinterConfig PDA
    Seeds:   ["minter", config.key(), minter_pubkey]
    Size:    90 bytes

    Fields:
    minter:              Pubkey   (32 bytes)  // The minter's public key
    stablecoin_config:   Pubkey   (32 bytes)  // Which stablecoin this minter belongs to
    quota:               u64      (8 bytes)   // Maximum amount allowed to mint (base units)
    minted:              u64      (8 bytes)   // Amount already minted (base units)
    active:              bool     (1 byte)    // Whether minting is currently allowed
    bump:                u8       (1 byte)    // PDA bump seed
```

Remaining quota is calculated as `quota.saturating_sub(minted)`. The master authority can:

- Add a new minter with a quota via `update_minter`.
- Increase an existing minter's quota.
- Deactivate a minter by setting `active = false`.

### Freeze Authority

The freezer role can freeze individual token accounts via the `freeze_account` instruction. A frozen account cannot send or receive tokens. The `thaw_account` instruction restores normal functionality. Freezing operates at the Token-2022 token account level, not the wallet level.

### On-Chain Metadata

Token metadata is stored in the mint account via the `MetadataPointer` extension:

| Field | Max Length | Description |
|-------|-----------|-------------|
| **Name** | 32 characters | Human-readable token name |
| **Symbol** | 10 characters | Ticker symbol |
| **URI** | 200 characters | URL pointing to off-chain JSON metadata |

### Role-Based Access Control

SSS-1 defines four operational roles:

| Role | Stored In | Capabilities |
|------|-----------|-------------|
| **Master Authority** | `StablecoinConfig.master_authority` | Update minters, update roles, transfer authority |
| **Pauser** | `RoleConfig.pauser` | Pause and unpause the stablecoin |
| **Freezer** | `RoleConfig.freezer` | Freeze and thaw individual token accounts |
| **Minter** | Individual `MinterConfig` PDAs | Mint tokens up to assigned quota |

At initialization, all roles (master authority, pauser, freezer) are assigned to the initializing wallet. The master authority should reassign roles to dedicated operational keypairs after setup.

The `RoleConfig` PDA also contains `blacklister` and `seizer` fields, but these roles have no effect in SSS-1. Any SSS-2-only instructions (`add_to_blacklist`, `remove_from_blacklist`, `seize`) will fail with `ComplianceNotEnabled` if called against an SSS-1 stablecoin.

### Pause / Unpause

The pauser can globally pause the stablecoin. When paused:

- `mint_tokens` reverts with `StablecoinError::Paused`.
- `burn_tokens` reverts with `StablecoinError::Paused`.
- Freeze, thaw, role management, and authority transfer operations continue to work.
- **SSS-1 transfers are NOT affected** by pausing. Since SSS-1 has no transfer hook, standard `transfer_checked` calls proceed normally even when the stablecoin is paused.

### Two-Step Authority Transfer

The master authority can be transferred via a two-step process to prevent accidental loss of control:

1. Current authority calls `transfer_authority(new_authority)` -- sets `pending_authority` in the config PDA.
2. New authority calls `accept_authority()` -- sets `master_authority = pending_authority` and clears `pending_authority`.

Only one pending transfer can exist at a time. The current authority retains full control until the new authority accepts. There is no explicit cancel instruction; the current authority can overwrite the pending authority by calling `transfer_authority` again with a different address (or their own address to effectively cancel).

### Burning

Any token holder can burn their own tokens via the `burn_tokens` instruction. This reduces the total supply. The stablecoin must not be paused. Burning is not restricted to any specific role.

## Lifecycle

```
           initialize
               |
               v
    +----------------------+
    | StablecoinConfig PDA  |
    | RoleConfig PDA        |
    | Mint (Token-2022)     |
    |   + MetadataPointer   |
    +----------------------+
               |
        update_minter (add minters with quotas)
               |
               v
    +----------------------+
    | MinterConfig PDA(s)   |
    | quota: 10,000,000     |
    +----------------------+
               |
          mint_tokens (minter signs, quota decremented)
               |
               v
    +----------------------+        freeze_account
    | Tokens in            | ------> Account frozen
    | circulation          | <------ thaw_account
    +----------------------+
               |
          burn_tokens (holder burns own tokens)
               |
               v
    +----------------------+
    | Supply reduced        |
    +----------------------+
```

### Step-by-Step

1. **Initialize**: Create the stablecoin with name, symbol, decimals, and URI. A fresh mint keypair is generated. The config PDA becomes the mint and freeze authority.

2. **Configure minters**: The master authority adds one or more minters via `update_minter`, each with a quota. This controls who can issue new tokens and how much.

3. **Mint tokens**: Authorized minters call `mint_tokens` to issue tokens to recipients. The minter's remaining quota is decremented by the minted amount. The program creates or uses the recipient's associated token account.

4. **Transfer tokens**: Holders transfer tokens using standard Token-2022 `transfer_checked`. No hook is involved in SSS-1, so transfers cannot be blocked by the stablecoin program.

5. **Freeze / Thaw**: The freezer can freeze individual token accounts. Frozen accounts cannot send or receive tokens. Thawing restores normal functionality.

6. **Burn tokens**: Any holder can burn their own tokens via `burn_tokens`, reducing the circulating supply. The config's `total_burned` counter is incremented.

7. **Pause / Unpause**: The pauser can halt all minting and burning globally.

## Account Structures

### StablecoinConfig (253 bytes)

```
Offset  Size  Field
0       8     Anchor discriminator
8       32    master_authority       (Pubkey)
40      32    pending_authority      (Pubkey, default = Pubkey::default())
72      32    mint                   (Pubkey)
104     1     decimals               (u8)
105     1     enable_permanent_delegate (bool, false for SSS-1)
106     1     enable_transfer_hook   (bool, false for SSS-1)
107     1     is_paused              (bool)
108     8     total_minted           (u64)
116     8     total_burned           (u64)
124     1     bump                   (u8)
125     128   _reserved              ([u8; 128])
```

For SSS-1, `enable_permanent_delegate` and `enable_transfer_hook` are both `false`. The `is_compliance_enabled()` method returns `false`.

### RoleConfig (169 bytes)

```
Offset  Size  Field
0       8     Anchor discriminator
8       32    stablecoin_config      (Pubkey)
40      32    pauser                 (Pubkey)
72      32    freezer                (Pubkey)
104     32    blacklister            (Pubkey, set but unused in SSS-1)
136     32    seizer                 (Pubkey, set but unused in SSS-1)
168     1     bump                   (u8)
```

### MinterConfig (90 bytes)

```
Offset  Size  Field
0       8     Anchor discriminator
8       32    minter                 (Pubkey)
40      32    stablecoin_config      (Pubkey)
72      8     quota                  (u64)
80      8     minted                 (u64)
88      1     active                 (bool)
89      1     bump                   (u8)
```

## Comparison to Other Standards

| Feature | SSS-1 | ERC-20 (Ethereum) | SPL Token (Legacy Solana) |
|---------|:-----:|:-----------------:|:-------------------------:|
| Mint authority | Yes | Varies | Yes |
| Freeze authority | Yes | No (by default) | Yes |
| On-chain metadata | Yes (Token-2022) | No (requires ERC-721-like) | No (requires Metaplex) |
| Role separation | Yes (4 roles) | No (single owner) | No |
| Per-minter quotas | Yes | No (typically) | No |
| Pause mechanism | Yes | Varies (OpenZeppelin) | No |
| Two-step authority transfer | Yes | No | No |

## Events

SSS-1 emits the following Anchor events. Each event is base64-encoded in the program's transaction logs and prefixed with the Anchor event discriminator.

| Event | Fields |
|-------|--------|
| `StablecoinInitialized` | config, mint, authority, enable_permanent_delegate, enable_transfer_hook, decimals, timestamp |
| `TokensMinted` | config, minter, recipient, amount, timestamp |
| `TokensBurned` | config, burner, amount, timestamp |
| `AccountFrozen` | config, account, frozen_by, timestamp |
| `AccountThawed` | config, account, thawed_by, timestamp |
| `StablecoinPaused` | config, paused_by, timestamp |
| `StablecoinUnpaused` | config, unpaused_by, timestamp |
| `MinterUpdated` | config, minter, quota, active, updated_by, timestamp |
| `RolesUpdated` | config, updated_by, timestamp |
| `AuthorityTransferInitiated` | config, current_authority, pending_authority, timestamp |
| `AuthorityTransferCompleted` | config, old_authority, new_authority, timestamp |

## Instructions

| Instruction | Signer | Description |
|-------------|--------|-------------|
| `initialize` | Authority + Mint Keypair | Create the stablecoin, config PDA, role config PDA, and mint |
| `mint_tokens` | Minter | Mint tokens to a recipient (quota enforced) |
| `burn_tokens` | Token holder | Burn tokens from the caller's account |
| `freeze_account` | Freezer | Freeze a token account |
| `thaw_account` | Freezer | Thaw a frozen token account |
| `pause` | Pauser | Globally pause minting and burning |
| `unpause` | Pauser | Resume minting and burning |
| `update_minter` | Master Authority | Add, update quota, or deactivate a minter |
| `update_roles` | Master Authority | Reassign pauser, freezer, blacklister, seizer roles |
| `transfer_authority` | Master Authority | Initiate two-step authority transfer |
| `accept_authority` | Pending Authority | Accept a pending authority transfer |

## Upgrading to SSS-2

SSS-1 and SSS-2 are initialization-time decisions. The preset is determined by the Token-2022 extensions configured on the mint at creation. Because Token-2022 extensions cannot be added to an existing mint, upgrading an SSS-1 stablecoin to SSS-2 requires creating a new mint with the SSS-2 preset and migrating token holders.

If there is any possibility that compliance features will be needed in the future, it is recommended to initialize with SSS-2 from the start. The compliance features can be left unused until needed.
