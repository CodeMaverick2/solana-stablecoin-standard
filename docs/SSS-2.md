# SSS-2: Compliant Stablecoin Standard

## Overview

SSS-2 is the compliant stablecoin preset in the Solana Stablecoin Standard. It extends SSS-1 with full regulatory compliance tooling: a permanent delegate authority for token seizure, a transfer hook program for blacklist and pause enforcement on every transfer, and PDA-based blacklist management with audit metadata.

SSS-2 is designed for regulated financial instruments that must comply with sanctions laws, court orders, and emerging legislation such as the GENIUS Act.

**Stablecoin Program ID**: `B1zqgaJkbVzNoMagPyAJdgveArzaTW6fkyk3JtSq1pHs`
**Transfer Hook Program ID**: `HgSUZDiLt8UWwzaxhCWwLPPs9zB1F7WTzCFSVmQSaLou`

---

## Use Cases

- **USDC/USDT-class stablecoins**: Fiat-backed stablecoins requiring regulatory compliance.
- **GENIUS Act compliance**: Meeting requirements for payment stablecoin issuers.
- **Regulated tokenized assets**: Securities, bonds, or real-world assets on Solana.
- **Central bank digital currency (CBDC) pilots**: Government-issued digital currency experiments.
- **Institutional settlement**: Cross-border payments requiring sanctions screening.

## Token-2022 Extensions

SSS-2 uses three Token-2022 extensions:

| Extension | Purpose |
|-----------|---------|
| **MetadataPointer** | On-chain metadata storage (same as SSS-1) |
| **PermanentDelegate** | Allows the config PDA to burn tokens from any account (seizure) |
| **TransferHook** | Invokes the transfer hook program on every `transfer_checked` call |

### MetadataPointer

Identical to SSS-1. Token metadata (name, symbol, URI) is stored directly in the mint account. The config PDA is the update authority.

### PermanentDelegate

The permanent delegate is set to the config PDA at mint initialization and **cannot be changed afterward**. This gives the stablecoin program the ability to burn tokens from any account without the account owner's signature, which is the mechanism used for token seizure.

The permanent delegate is controlled by the program, not by any individual. Only the designated **seizer** role can invoke the `seize` instruction, which uses the permanent delegate authority.

### TransferHook

The transfer hook program (`HgSUZDiLt8UWwzaxhCWwLPPs9zB1F7WTzCFSVmQSaLou`) is configured on the mint at initialization. Token-2022 invokes this program on every `transfer_checked` call. The hook performs three checks:

1. **Pause check**: Is the stablecoin globally paused?
2. **Sender blacklist check**: Is the sender's wallet address blacklisted?
3. **Recipient blacklist check**: Is the recipient's wallet address blacklisted?

If any check fails, the transfer is rejected. There is no way to bypass the hook -- it is enforced by the Token-2022 program itself at the runtime level.

## Initialization

### Mint Creation

Like SSS-1, the mint is a fresh `Keypair` generated by the client. The program uses raw CPI for mint creation because the Token-2022 extensions must be configured dynamically. For SSS-2, the extension initialization order is:

1. `create_account` -- allocate space for the mint with all three extensions.
2. `initialize_metadata_pointer` -- point the metadata to the mint account itself.
3. `initialize_permanent_delegate` -- set the config PDA as permanent delegate.
4. `initialize_transfer_hook` -- set the transfer hook program on the mint.
5. `initialize_mint2` -- initialize the mint with the config PDA as mint authority and freeze authority.
6. `initialize_token_metadata` -- write name, symbol, and URI.

### ExtraAccountMetaList Initialization

After the stablecoin is initialized, the `ExtraAccountMetaList` PDA must be initialized on the transfer hook program before any transfers can succeed. This is a separate instruction (`initialize_extra_account_meta_list`) on the transfer hook program.

The SDK's `SolanaStablecoin.create()` method handles this automatically by sending both the `initialize` instruction and the `initialize_extra_account_meta_list` instruction.

```
ExtraAccountMetaList PDA
    Seeds:   ["extra-account-metas", mint.key()]
    Program: Transfer Hook (HgSUZDiLt8UWwzaxhCWwLPPs9zB1F7WTzCFSVmQSaLou)
```

### Accounts Created

| Account | Type | Seeds | Program |
|---------|------|-------|---------|
| **Mint** | Token-2022 Mint | N/A (Keypair) | Token-2022 |
| **StablecoinConfig** | PDA | `["stablecoin_config", mint.key()]` | Stablecoin |
| **RoleConfig** | PDA | `["roles", config.key()]` | Stablecoin |
| **ExtraAccountMetaList** | PDA | `["extra-account-metas", mint.key()]` | Transfer Hook |

### SDK / CLI

```typescript
// SDK
const { sdk, mintKeypair } = await SolanaStablecoin.create(

```

```bash

```

## Features (SSS-1 Inherited)

SSS-2 includes all SSS-1 capabilities:

- Mint authority with per-minter quotas
- Freeze authority
- On-chain metadata (MetadataPointer)
- Role-based access control (master authority, pauser, freezer, minters)
- Pause / unpause
- Two-step authority transfer
- Self-burn by token holders

See [SSS-1.md](SSS-1.md) for full documentation on these features.

**Key difference from SSS-1**: When the stablecoin is paused, SSS-2 transfers are **also blocked** by the transfer hook. In SSS-1, pausing only affects minting and burning; transfers proceed normally.

## SSS-2 Specific Features

### Additional Roles

SSS-2 adds two compliance-specific roles:

| Role | Stored In | Capabilities |
|------|-----------|-------------|
| **Blacklister** | `RoleConfig.blacklister` | Add and remove addresses from the blacklist |
| **Seizer** | `RoleConfig.seizer` | Seize tokens from any account via burn + mint |

Full role table for SSS-2:

| Role | Operations |
|------|-----------|
| Master Authority | Manage minters, update roles, transfer authority |
| Pauser | Pause / unpause |
| Freezer | Freeze / thaw accounts |
| Minter | Mint tokens (up to quota) |
| Blacklister | Add / remove blacklist entries |
| Seizer | Seize tokens to treasury |

### Blacklist Management

The blacklist uses individual PDA accounts per blacklisted address:

```
BlacklistEntry PDA
    Seeds:   ["blacklist", config.key(), address]
    Program: Stablecoin (B1zqgaJkbVzNoMagPyAJdgveArzaTW6fkyk3JtSq1pHs)

    Fields:
    stablecoin_config:   Pubkey   (32 bytes)
    address:             Pubkey   (32 bytes)  // The blacklisted wallet
    reason:              String   (4 + up to 128 bytes)
    blacklisted_at:      i64      (8 bytes)   // Unix timestamp
    blacklisted_by:      Pubkey   (32 bytes)  // Who blacklisted them
    bump:                u8       (1 byte)
```

**Why PDAs instead of a bitmap?**

| PDA Approach | Bitmap Approach |
|-------------|----------------|
| O(1) existence check via `data_is_empty()` | O(1) bit check |
| Each entry stores metadata (reason, timestamp, who) | No metadata per entry |
| Individually closeable (rent recovery) | Cannot recover rent for individual entries |
| No size limits on the blacklist | Fixed size, must be resized |
| Higher per-entry cost (~0.002 SOL rent) | Lower per-entry cost |

For compliance use cases, the metadata stored per entry (reason, who, when) is critical for audit trails and regulatory reporting.

**Adding to blacklist**:
- The blacklister calls `add_to_blacklist` with the wallet address and a reason string (max 128 characters).
- The program creates the `BlacklistEntry` PDA.
- The transfer hook immediately begins rejecting transfers involving this address.
- An `AddedToBlacklist` event is emitted.

**Removing from blacklist**:
- The blacklister calls `remove_from_blacklist` with the wallet address.
- The program closes the `BlacklistEntry` PDA and returns rent to the payer.
- The transfer hook immediately stops rejecting transfers for this address.
- A `RemovedFromBlacklist` event is emitted.

### Transfer Hook Enforcement

Every `transfer_checked` call goes through the transfer hook. Token-2022 resolves the `ExtraAccountMetaList` for the mint, then CPIs into the transfer hook program with the resolved accounts.

```
Token-2022 transfer_checked
    |
    +-- Resolves ExtraAccountMetaList for the mint
    |
    +-- CPIs into transfer hook program
    |       |
    |       +-- assert_is_transferring()  --> Verify legitimate transfer context
    |       +-- Is stablecoin paused?     --> Reject (StablecoinPaused)
    |       +-- Is sender blacklisted?    --> Reject (SenderBlacklisted)
    |       +-- Is recipient blacklisted? --> Reject (RecipientBlacklisted)
    |
    +-- Transfer completes (only if all checks pass)
```

#### ExtraAccountMetaList

The ExtraAccountMetaList stores five additional accounts that Token-2022 resolves and passes to the hook. The first five accounts (indices 0-4) are mandated by the Transfer Hook interface:

| Index | Account | Description |
|-------|---------|-------------|
| 0 | Source Token Account | The sender's token account |
| 1 | Mint | The stablecoin mint |
| 2 | Destination Token Account | The recipient's token account |
| 3 | Owner | The source token account authority |
| 4 | ExtraAccountMetaList | The PDA storing extra account metadata |

Indices 5-9 are the custom extra accounts defined by the stablecoin:

| Index | Account | Type | Purpose |
|-------|---------|------|---------|
| 5 | Stablecoin Config | Static pubkey | Read `is_paused` flag |
| 6 | Stablecoin Program | Static pubkey | Program ID for blacklist PDA derivation |
| 7 | Sender Blacklist Entry | External PDA | Derived from source token owner via stablecoin program |
| 8 | Recipient Blacklist Entry | External PDA | Derived from dest token owner via stablecoin program |
| 9 | Transfer Hook Program | Static pubkey | Self-reference for PDA derivation context |

**Account ordering is critical.** The stablecoin program (index 6) must come before the blacklist PDAs (indices 7 and 8) because the blacklist PDAs use `new_external_pda_with_seeds` with `account_index: 6` to specify the program that owns the blacklist PDAs.

The blacklist entry PDAs at indices 7 and 8 are derived dynamically:
- **Seeds**: `["blacklist", config_key (from index 5), owner_pubkey (from token account data at offset 32)]`
- **Program**: stablecoin program (index 6)

The owner pubkey is extracted from the token account data using `Seed::AccountData` at `data_index: 32` (the owner field offset in a Token-2022 account) with `length: 32` (pubkey size).

#### Pause Check (Raw Byte Offset)

The transfer hook reads the `is_paused` field directly from the `StablecoinConfig` account data at byte offset 107:

```
Offset 107 = 8 (discriminator) + 32 (master_authority) + 32 (pending_authority)
           + 32 (mint) + 1 (decimals) + 1 (enable_permanent_delegate)
           + 1 (enable_transfer_hook) = 107
```

If the byte at offset 107 is non-zero, the stablecoin is paused and the transfer is rejected with `TransferHookError::StablecoinPaused`.

This raw byte read is used instead of Anchor deserialization for gas efficiency. The transfer hook does not need to deserialize the entire config account -- it only needs one byte.

#### Blacklist Check

The transfer hook checks blacklist status by calling `data_is_empty()` on the sender and recipient blacklist entry accounts:
- If the account has data (i.e., the PDA exists and is initialized), the address is blacklisted.
- If the account is empty (PDA does not exist or has been closed), the address is not blacklisted.

This is an O(1) check with no deserialization overhead.

#### Anti-Spoofing: assert_is_transferring()

The `assert_is_transferring()` function prevents the transfer hook from being called directly by an attacker. It reads the `TransferHookAccount` extension from the source token account and checks the `transferring` flag. This flag is only set by Token-2022 during an active `transfer_checked` call. If the flag is not set, the hook rejects the call with `IsNotCurrentlyTransferring`.

#### Fallback Handler

The transfer hook includes a `fallback` function that bridges Anchor's 8-byte instruction discriminators with the SPL `TransferHookInstruction` format. Token-2022 calls the hook using the SPL format, so the fallback parses the SPL instruction data and dispatches to the Anchor `transfer_hook` handler.

### Token Seizure

The `seize` instruction uses a **burn + mint** pattern rather than `transfer_checked`. This is a deliberate design decision:

```rust
// Step 1: Burn tokens from the source account using permanent delegate
invoke_signed(
    &spl_token_2022::instruction::burn_checked(
        &spl_token_2022::ID,
        &from_token_account,
        &mint,
        &config_pda,        // permanent delegate signs
        &[],
        amount,
        decimals,
    )?,
    accounts,
    &[config_pda_seeds],
)?;

// Step 2: Mint equivalent tokens to the treasury using mint authority
invoke_signed(
    &spl_token_2022::instruction::mint_to(
        &spl_token_2022::ID,
        &mint,
        &to_token_account,
        &config_pda,        // mint authority signs
        &[],
        amount,
    )?,
    accounts,
    &[config_pda_seeds],
)?;
```

**Why burn + mint instead of transfer_checked?**

1. **Bypasses the transfer hook**: A `transfer_checked` call would trigger the transfer hook, which would reject the transfer because the source account is likely blacklisted. The burn + mint pattern avoids this entirely.
2. **Works on frozen accounts**: The permanent delegate can burn tokens from a frozen account. Token-2022's freeze enforcement does not apply to the permanent delegate's burn authority.
3. **Supply-neutral**: The burn and mint cancel out, so the total token supply is unaffected.
4. **Config PDA serves dual role**: The config PDA is both the permanent delegate (for the burn) and the mint authority (for the mint), so a single PDA signer seeds array suffices for both operations.

**Requirements**:
- Caller must have the **seizer** role.
- Stablecoin must have `enable_permanent_delegate = true` (checked via `is_compliance_enabled()`).
- Amount must be greater than zero.
- Both source and destination must be valid token accounts for this mint.

## Compliance Flow

The full compliance lifecycle for a flagged address:

```
1. Sanctions screening identifies a match
          |
          v
2. sss-token blacklist add <address> --reason "OFAC match"
   --> Creates BlacklistEntry PDA
   --> Emits AddedToBlacklist event
   --> All future transfers involving this address are rejected by the hook
          |
          v
3. sss-token freeze <token-account>
   --> Freezes the token account at the Token-2022 level
   --> Emits AccountFrozen event
   --> Account cannot participate in any token operations
          |
          v
4. sss-token seize <token-account> --to <treasury>
   --> Burns tokens from source (permanent delegate), mints to treasury
   --> Emits TokensSeized event
   --> Tokens moved to treasury; supply unchanged
          |
          v
5. Record all transaction signatures for audit trail
          |
          v
6. (If cleared later)
   sss-token blacklist remove <address>
   sss-token thaw <token-account>
```

| Feature | SSS-1 | SSS-2 |
|---------|:-----:|:-----:|

## Security Considerations

### Transfer Hook Cannot Be Bypassed

The transfer hook is enforced by Token-2022 at the runtime level. Any `transfer_checked` call on an SSS-2 mint will invoke the hook. There is no way to call `transfer` (without the `_checked` suffix) on Token-2022 mints that have a transfer hook configured.

### Permanent Delegate Cannot Be Changed

Once set at mint initialization, the permanent delegate cannot be modified. This means the config PDA will always be the permanent delegate for the lifetime of the mint. This is a Token-2022 design constraint.

### Seize Works on Frozen Accounts

The burn + mint pattern works on frozen accounts because the permanent delegate's burn authority is not subject to the freeze flag. This is intentional: a seized account is typically frozen first, and the seizure must still be executable.

### Blacklist Does Not Block Minting

Blacklisting an address prevents transfers via the transfer hook, but minting to a blacklisted address is still possible because `mint_to` does not invoke the transfer hook. To fully restrict an address, also freeze their token account.
